# Session API

## Why `session`

A `session` is the top-level optimization lifecycle:
- one generated `sessionId`
- one fixed trajectory set
- multiple candidate versions
- multiple candidate runs
- one acceptance loop

## Phase 1: Create Session

API:

```ts
createSession(config: SessionConfig): Promise<Session>
```

Input:

```ts
type SessionConfig = {
  maxIterations: number;

  // Optional pass-rate target that allows early stopping.
  // If a checkpoint reaches this threshold, the session can end.
  acceptanceThreshold?: number;
};
```

Output:

```ts
type Session = {
  sessionId: string;

  // Immutable copy of the rules that govern the session.
  // Keeping the config on the session makes later decisions auditable.
  config: SessionConfig;
  createdAt: string;
};
```

Notes:
- `sessionId` is auto-generated by the system.
- This phase exists so the optimizer has one root object that owns the full workflow.

## Phase 2: Create Trajectory Set

API:

/*
<Trajectory> :   `Trajectory` is a generic placeholder type.
The caller provides the concrete trajectory shape so this API can stay reusable
while still type-checking the `trajectories` array.
*/


```ts
createTrajectorySet<Trajectory>(
  sessionId: string,                   //links the set to the correct session 
  trajectories: readonly Trajectory[]
): Promise<TrajectorySet>
```

Input:

```ts
type CreateTrajectorySetInput<Trajectory> = {
  // Links the trajectory set to the session that will use it.
  sessionId: string;
  trajectories: readonly Trajectory[];
};
```

Output:

```ts
type TrajectorySet = {
  // Records which trajectories were included without copying the
  // trajectory payload back into this contract.
  trajectoryIds: string[];
  createdAt: string;
};
```

Notes:
- `sessionId` is required here because the trajectory set belongs to a specific session.
- Once the set is created, candidate quality can be compared fairly.


## Phase 3: Create Candidate Version

API:

```ts
createCandidateVersion(input: CreateCandidateVersionInput): Promise<CandidateVersion>
```

Input:

```ts
type CreateCandidateVersionInput = {
  // The checkpoint anchors this operation to the currently known state:
  // which candidate was evaluated and with what score.
  checkpoint: Checkpoint;

  // Structured output from failure analysis that tells generation what needs to improve next.
  analysis: FailureAnalysis;
};
```

Output:

```ts
type CandidateVersion = {
  // Unique identifier for one generated candidate version.
  // This should identify the candidate instance directly.
  candidateId: string;

  // Creation time for ordering candidate versions.
  // The latest candidate can be identified by this timestamp.
  createdAt: string;
};

```

Notes:
- `checkpoint` already identifies the candidate the next version is derived from.

## Shared Evaluation Summary Type

/*
EvalName extends string = string: 
`EvalName` must be some kind of string because eval identifiers are names.
Callers can narrow it to specific values like `"safety" | "accuracy"`,
and `= string` makes plain string the default when nothing is provided.
*/

```ts
type ScopeIssue<
  EvalName extends string = string
> = {
  // Eval that produced the issue.
  eval: EvalName;

  // Human-readable explanation of what needs attention.
  reason: string;

  // Optional pass rate copied from the eval summary when relevant.
  passRate: number;
};

type ScopeOverview<EvalName extends string = string> = {
  // One synthesized summary across all evals in the scope.
  summary: string;

  // Issues collected from the scope's per-eval summaries.
  issues: ScopeIssue<EvalName>[];

  // Evals currently failing or below the desired threshold.
  failingEvals: EvalName[];

  // Evals currently healthy in this scope.
  passingEvals: EvalName[];
};

type EvalSummaries<
  SingleTurnEvalName extends string = string,
  MultiTurnEvalName extends string = string
> = {
  // Per-eval summaries for step-level evaluations.
  singleTurn: Record<SingleTurnEvalName, EvalSummary>;

  // Per-eval summaries for conversation-level evaluations.
  multiTurn: Record<MultiTurnEvalName, EvalSummary>;

  // Rolled-up issue summary across all single-turn eval summaries.
  singleTurnOverview: ScopeOverview<SingleTurnEvalName>;

  // Rolled-up issue summary across all multi-turn eval summaries.
  multiTurnOverview: ScopeOverview<MultiTurnEvalName>;
};
```


## Phase 4: Create Candidate Run

API:

```ts
createCandidateRun<Trajectory>(
  input: CreateCandidateRunInput
): Promise<CandidateRun<Trajectory>>
```

Input:

```ts
type CreateCandidateRunInput = {
  // Session scope ensures the run uses the correct trajectory set and rules.
  sessionId: string;

  // The candidate version to execute against the session's trajectory set.
  candidate: CandidateVersion;
};
```

Output:

```ts
type CandidateRun<Trajectory> = {
  // Owning session for this execution batch.
  sessionId: string;

  // Candidate identifier for the exact candidate that was executed.
  candidateId: string;

  // The trajectories executed in this run.
  // The trajectory layer owns the concrete trajectory shape.
  trajectories: readonly Trajectory[];

  // Lifecycle timing for monitoring and ordering.
  startedAt: string;
  completedAt?: string;
};
```

## Phase 5: Evaluate Candidate Run

API:

```ts
evaluateCandidateRun<Trajectory>(
  input: EvaluateCandidateRunInput<Trajectory>
): Promise<CandidateRunEvaluation>
```

Input:

```ts
type EvaluateCandidateRunInput<Trajectory> = {
  // The completed execution batch being scored.
  run: CandidateRun<Trajectory>;
};
```

Output:

```ts
type CandidateRunEvaluation = {
  // Session scope for consistency checks.
  sessionId: string;

  // Candidate identifier for the exact candidate that was evaluated.
  candidateId: string;

  // Tally summaries split by eval scope, plus rolled-up issue summaries
  // for single-turn and multi-turn evaluation groups.
  evalSummaries: EvalSummaries;

  // Single session-level score used for optimization decisions.
  // This compresses many eval outputs into one decision signal.
  aggregatedPassRate: number;
};
```

Notes:
- Evaluation should be derived from Tally reports via `report.view()`.
- `singleTurn` and `multiTurn` preserve per-eval summaries.
- `singleTurnOverview` and `multiTurnOverview` are derived from those per-eval summaries.
- `evalSummaries` is the main comparison surface.
- This phase translates raw run artifacts into optimizer decision data.
- The optimizer should compare evaluations, not low-level step outputs.

## Phase 6: Create Checkpoint

API:

```ts
createCheckpoint(
  sessionId: string,
  candidate: CandidateVersion,
  evaluation: CandidateRunEvaluation
): Promise<Checkpoint>
```

Input:

```ts
type CreateCheckpointInput = {
  // Owning session for the checkpoint.
  sessionId: string;

  // Candidate version being checkpointed.
  candidate: CandidateVersion;

  // Evaluation snapshot that justifies the checkpoint.
  evaluation: CandidateRunEvaluation;
};
```

Output:

```ts
type Checkpoint = {
  // Unique identifier for this saved optimizer state.
  checkpointId: string;

  // Session that owns the checkpoint.
  sessionId: string;

  // Candidate identifier for the exact candidate captured in this checkpoint.
  candidateId: string;

  // Stored Tally artifacts that back this checkpoint.
  // Each item comes from a completed trajectory run after the system calls
  // `report.toArtifact()` on that trajectory's `TallyRunReport`.
  // These are needed for replay, debugging, or later inspection.
  tallyArtifacts: Array<{
    trajectoryId: string;
    runId: string;
    artifactPath: string;
  }>;

  // Eval snapshots used for later comparison, including per-eval detail
  // and scope-level issue rollups.
  evalSummaries: EvalSummaries;

  // Top-level score saved with the checkpoint so acceptance logic
  // does not need to recompute it.
  aggregatedPassRate: number;

  // Optional acceptance result once this checkpoint has been judged
  // against a previous checkpoint.
  accepted?: boolean;
  rejectReason?: string;

  // Creation timestamp for ordering and history.
  createdAt: string;
};
```

Notes:
- A checkpoint is the optimizer's durable decision boundary.
- It is the object you compare, accept, reject, or analyze later.

## Phase 7: Analyze Failures

API:

```ts
analyzeFailures(
  input: AnalyzeCheckpointFailuresInput
): Promise<FailureAnalysis>
```

Input:

```ts
type AnalyzeCheckpointFailuresInput = {
  // The saved optimizer state being inspected.
  checkpoint: Checkpoint;
};
```

Output:

```ts
type FailureAnalysis = {
  // Concrete failures extracted from Tally outputs.
  // These become the evidence for the next mutation step.
  failures: Array<{
    trajectoryId: string;
    eval: string;
    level: "step" | "conversation" | "summary";
    reason?: string;
  }>;

  // High-value regions of the candidate content to change next.
  // This keeps generation focused instead of rewriting everything.
  targetBlocks: string[];
};
```

Notes:
- Prefer explicit Tally verdict failures first.
- Otherwise use low pass-rate eval summaries or scope overview issues.
- This phase turns score data into actionable change guidance.

## Phase 8: Evaluate Acceptance

API:

```ts
evaluateAcceptance(
  previous: Checkpoint,
  current: Checkpoint,
  options?: AcceptanceOptions
): Promise<AcceptanceDecision>
```

Input:

```ts
type AcceptanceOptions = {
  // Optional allowlist of evals that must be present and considered
  // during the acceptance decision.
  requiredEvals?: string[];
};

type EvaluateAcceptanceInput = {
  // Previously accepted or baseline checkpoint.
  previous: Checkpoint;

  // Newly produced checkpoint under consideration.
  current: Checkpoint;

  // Extra policy controls for the decision.
  options?: AcceptanceOptions;
};
```

Output:

```ts
type AcceptanceDecision = {
  // Final accept / reject result.
  accepted: boolean;

  // Human-readable explanation of the decision.
  reason: string;

  // Structured checks so callers know exactly what passed or failed.
  checks: {
    passRateImproved: boolean;
    sameSession: boolean;
    requiredEvalsPresent: boolean;
  };
};
```

Notes:
- This phase exists to keep acceptance logic explicit and auditable.
- The decision should never rely on intuition alone; it should point to named checks.

## Phase 9: Evaluate Stop Condition

API:

```ts
evaluateStopCondition(input: StopConditionInput): StopDecision
```

Input:

```ts
type StopConditionInput = {
  // Current loop number.
  iteration: number;

  // Latest checkpoint being considered.
  checkpoint: Checkpoint;

  // Hard cap for the session.
  maxIterations: number;

  // Optional early-stop target from the session config.
  acceptanceThreshold?: number;
};
```

Output:

```ts
type StopDecision = {
  // Whether the optimizer should exit the loop now.
  stop: boolean;

  // Explicit stop reason so callers can branch correctly.
  reason: "thresholdReached" | "maxIterations";
};
```

Notes:
- This phase separates loop-control policy from execution logic.
- It gives the optimizer one clear place to decide whether to continue.
